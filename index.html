Content Refresh Opportunity Scoring

Prioritizing content for editorial review using observable search and engagement signals

Abstract

Content teams managing large libraries need a repeatable way to decide which pages deserve review first. This project frames that decision as a ranking problem: given search and engagement signals for 30,000 content items, produce a prioritized queue with reason codes a human reviewer can act on. A rule-based baseline score (staleness, visibility, and click-through rate) was built first, then compared against a Random Forest classifier trained on the same features and evaluated under both a random split and a client-grouped split. Both approaches agreed closely on which items were highest priority, with staleness (days_since_last_update) dominating the model's feature importance. The output is a ranked action queue with reason codes, intended strictly as decision support for human reviewers — not as an automated publishing or ranking-prediction system.

Introduction

The decision this work supports is: which content pages should be prioritized for review and potential refresh, based on their search performance, engagement, and content freshness. The ranked recommendations are intended for content managers, SEO specialists, or digital marketing teams, who use them to decide whether to refresh, expand, rewrite, monitor, or leave a page unchanged.

Getting this wrong has asymmetric costs. A false positive — flagging a healthy page for review — wastes reviewer time. A false negative — missing a page that is genuinely declining — lets traffic and engagement erode further before anyone intervenes. Because impressions, clicks, CTR, average position, content age, and engagement interact in ways that are hard to capture in a single fixed rule, this project explores whether a learned model can rank pages more effectively than a manual rule — while keeping a human in the loop for every action.

Data

The analysis uses an anonymized content-performance dataset of 30,000 content items across 44 features, covering search volume, competition, content type, word/character counts, 90-day impressions, clicks, sessions, engagement rate, content age, freshness, click-through rate, average search position, and trend direction. One row represents one content item.

Descriptive figures from the initial data pull:

  • Content age ranged from 90 to 564 days, with a median of 236 days.
  • 90-day impressions were highly skewed — median of 731, but a long tail up to 517,715.
  • Trend direction was mostly down (16,262 items), followed by stable (5,962), up (4,388), new (2,236), and flat (1,152).

Excluded from features: client_id and content_id (identifiers only), and trend_direction / trend_pct (excluded due to leakage risk — these fields are derived in a way that could reflect information not available at prediction time; see Leakage Audit below).

No client names, domains, URLs, or private query data appear anywhere in this analysis or in the underlying repository, per the dataset's public-use terms.

Methodology

Baseline rule

The baseline scores each item using three binary signals:

  • Stale: days_since_last_update >= 180
  • Visible: impressions_90d >= 500
  • Low CTR: ctr < 1

baseline_score = 40×stale + 30×visible + 30×low_ctr, producing a 0–100 score with reason codes STALE_LOWCTR, STALE_VISIBLE, LOWCTR_VISIBLE, and REVIEW_PRIORITY as a default.

Model

A Random Forest classifier was trained on three features — days_since_last_update, impressions_90d, and ctr — to predict a high_priority label (baseline_score >= 70). This lets the model be compared directly against the rule it was trained to approximate, using an 80/20 split with a fixed random seed (42) for reproducibility.

Validation design

Because multiple content items can share a client, a plain random split risks leaking information about a client's typical behavior between train and test sets. The model was therefore re-evaluated under a client-grouped split (GroupShuffleSplit on client_id), so that no client appears in both the training and test partitions.

Honest caveat: the high_priority target was derived directly from the same rule (baseline_score) used to build the model's input features. The near-perfect accuracy reflects the model successfully recovering a deterministic rule from its own inputs — it is not evidence that the model predicts an independent, real-world outcome. This is a demonstration that a learned model can reproduce and generalize a scoring rule across a client-grouped split, not a claim of predictive power beyond that rule.

Limitations & honest framing

This analysis identifies observed patterns, not causal relationships. A high refresh score does not guarantee that updating a page will improve its performance. The results should be used as decision support alongside human review — never as an automated publishing, deletion, or metadata-change trigger. This work does not claim to identify Google ranking factors, does not predict future search performance, and does not establish that refreshing content causes performance gains.

Additional limits:

  • The evaluation window covers a single dataset snapshot; seasonal or temporary effects are not separated out.
  • The high_priority label is a proxy derived from the baseline rule, not an independent outcome measure (see Results note above).
  • Missing values in fields like word_count were not imputed or modeled explicitly.
  • Recommendations reflect the state of the data at pull time; recently updated content may not yet be reflected.

Ranked recommendations

The action playbook maps baseline_score to three tiers:

ScoreAction
≥ 70Refresh Content
40–69Review Content
< 40Monitor

Each item also carries a reason code (Stale Content, Low CTR, Stale + High Impressions + Low CTR, or Healthy Performance) so reviewers can see why an item was flagged, not just that it was.

Human review checklist

Before acting on any recommendation, a reviewer should verify:

  • Search intent still matches the topic.
  • Content is factually accurate.
  • Recent business priorities are considered.
  • Important pages are not removed accidentally.

No-go list — never automated

  • Publishing new content
  • Deleting pages
  • Rewriting content
  • Changing SEO metadata
  • Making business decisions without human approval

Monitoring / retrain triggers

The scoring system should be revisited if:

  • CTR changes substantially across the portfolio.
  • Search rankings shift materially.
  • New content types are introduced.
  • Model performance on fresh data decreases.
  • Business priorities change.
